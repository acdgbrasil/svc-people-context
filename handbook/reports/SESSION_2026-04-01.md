# Session Report — 2026-04-01/02

## Summary

Full design, contracting, and implementation session for two new ACDG services: **queue-manager** (contracts only) and **people-context** (contracts + full implementation + deploy).

---

## Decisions Made

### queue-manager
- **Stack**: Dart AOT + shelf (HTTP) + PostgreSQL + NATS
- **Domain**: 5 bounded contexts (Triage, Queues, Attendance, Panels, Admin) fully documented in handbook
- **Triage split**: queue-manager owns the operational queue; social-care owns the social assessment. Frontend orchestrates both.

### people-context
- **Stack**: Bun 1.3.11 + Elysia 1.4.28 + PostgreSQL 15 + NATS + jose (JWT)
- **Scope**: Minimal identity registry — Person (fullName, cpf, birthDate) + SystemRoles per service. No domain data (address, family, specialty lives in each consuming service).
- **Principle**: "Everything a system needs to operate lives inside that system. People Context only registers that a person exists."
- **Auth integration**: People does NOT talk to Zitadel. Future IAM Service will handle user provisioning.
- **Patients will have login**: created after triage via future IAM Service.

### TypeScript conventions
- 100% functional — no classes, no `this`, no `any`
- `type` over `interface` (per TS handbook reference)
- Branded types for IDs (`PersonId`, `Cpf`, `RoleId`)
- Discriminated unions for validation results
- `unknown` instead of `any` — always narrow

## Security Audit

All private cloud security directives implemented:
- JWT RS256 via Zitadel JWKS (jose library)
- RBAC per endpoint (social_worker for writes, social_worker/owner/admin for reads, admin for role lifecycle)
- Token introspection fallback (RFC 7662) for service accounts with explicit allowlist
- X-Actor-Id enforcement on all mutations
- Parameterized SQL (postgres.js tagged templates)
- Secrets via Bitwarden Secrets Manager (BitwardenSecret CRDs)
- Database-per-service isolation
