# 02 — Aggregates & Model

## Overview

The People Context has intentionally minimal domain model — two entities, no complex business logic. The complexity lives in the consuming services, not here.

---

## Aggregate: Person

The root entity. Represents the **existence** of a person in the ACDG ecosystem.

```
Person {
  id:        PersonId (UUID, generated server-side)
  fullName:  string (min 1, max 200)
  cpf:       string (11 digits, unique, nullable)
  birthDate: date (ISO 8601)
  createdAt: datetime
  updatedAt: datetime
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | UUID | Yes (auto) | Canonical person identifier used across all ACDG systems. Generated by the People Context on registration. |
| `fullName` | string | Yes | Full display name. Used by consuming services for UI (panels, reports, receipts). |
| `cpf` | string (11 digits) | No | Brazilian CPF. When present, serves as the **deduplication key** — no two persons can share the same CPF. |
| `birthDate` | date | Yes | Date of birth. Used for disambiguation of homonyms and age-based logic in consuming services. |
| `createdAt` | datetime | Yes (auto) | When the person was first registered. |
| `updatedAt` | datetime | Yes (auto) | Last modification timestamp. |

### Invariants

1. **CPF uniqueness**: If a CPF is provided, no other Person can have the same CPF. Registration with a duplicate CPF returns the existing Person (idempotent dedup).
2. **PersonId is immutable**: Once generated, a PersonId never changes. All systems reference this ID permanently.
3. **fullName is not empty**: A person must have a displayable name.
4. **birthDate is not in the future**: Self-explanatory.

### Lifecycle

```
[Register] → Person created (PersonId generated)
[Update]   → fullName, cpf, birthDate can be updated
[Dedup]    → Register with existing CPF returns existing Person (no duplicate created)
```

There is no "delete" — persons are permanent records in the ecosystem. Systems can deactivate their own roles, but the Person identity persists.

## Entity: PersonSystemRole

Tracks **what a person is in each system**. This is not auth (that's Zitadel) — it's domain identity.

```
PersonSystemRole {
  id:         UUID (auto)
  personId:   PersonId (FK → Person)
  system:     string (service identifier)
  role:       string (domain role in that system)
  active:     boolean (default true)
  assignedAt: datetime (auto)
}
```

### Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | UUID | Yes (auto) | Role assignment identifier. |
| `personId` | UUID | Yes | References the Person. |
| `system` | string | Yes | Service identifier. Values: `social-care`, `queue-manager`, `therapies`, `timesheet`, etc. |
| `role` | string | Yes | Domain role in that system. Values depend on the system (see below). |
| `active` | boolean | Yes | Whether this role is currently active. |
| `assignedAt` | datetime | Yes (auto) | When the role was assigned. |

### Known system × role combinations

| System | Possible Roles | Assigned when |
|--------|---------------|---------------|
| `social-care` | `patient`, `family-member` | Triage conclusion / family member registration |
| `queue-manager` | `patient`, `professional` | Arrival at unit / admin registers professional |
| `therapies` | `patient`, `therapist` | Therapy enrollment / therapist registration |
| `timesheet` | `employee` | Employee onboarding |

### Invariants

1. **No duplicate active roles**: A person cannot have two active roles with the same (system, role) combination.
2. **Person must exist**: The referenced PersonId must exist in the Person table.
3. **System and role are not empty**: Both fields must be non-blank strings.

### Lifecycle

```
[Assign]     → New role created (active=true)
[Deactivate] → Role set to active=false (soft removal)
[Reactivate] → Role set back to active=true
```

## What is NOT in this model

| Concept | Where it lives instead |
|---------|----------------------|
| Address, phone, email | Each consuming service |
| Family relationships (FamilyGroup) | social-care |
| Professional specialty, CRP/CREFITO | queue-manager, therapies |
| Work schedule, shift hours | timesheet |
| Diagnoses, assessments | social-care |
| Auth credentials, login | Zitadel (via future IAM Service) |
| Queue position, service orders | queue-manager |
| Therapy sessions | therapies |

## Events

The People Context publishes events when its state changes:

| Event | Trigger | Payload |
|-------|---------|---------|
| `people.person.registered` | New Person created | personId, fullName, cpf, birthDate |
| `people.person.updated` | Person data changed | personId, changed fields |
| `people.role.assigned` | New system role added | personId, system, role |
| `people.role.deactivated` | Role set to inactive | personId, system, role |
| `people.role.reactivated` | Role reactivated | personId, system, role |

These events allow consuming services to react (e.g., social-care could listen for `people.person.updated` to refresh cached names).
